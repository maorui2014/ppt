1.1、需求管理

目前从实施和客服接过来的需求文档有如下几点问题：

  • 需求文档不规范
  • 需求点描述不到位
  • 需求理解偏差（极少）
  • 需求交付时间长，质量低
  • 需求归档不够到位

解决方案

追根溯源，一切工作要从需求开始，也必须到需求结束

  • 实施或客服需与客户共同制定需求文档
  • 成立需求审核小组
  • 与客户共同定制合理的交付时间
  • 设立专门的文档管理员

1.2、与实施、客服的对接

公司从年初就定了这种工作模式，实际上形同虚设。

实施与开发貌合神离，互不理解，互相埋怨

解决方案

实施、客服与开发绑定为一个团队，荣辱与共！

- 设立团队责任人 -

实施的管理模式

  • 实施的管理是基于项目的，以项目实施计划为基准
  • 制定项目关键节点、回款等考核目标
  • 保障实施人员工作完成度，同时保障实施项目进度质量

遇到的问题

实施遇到的问题分为内部与外部：

  • 内部主要是系统稳定性和功能设计问题，导致BUG和需求产生，由于开发资源有限，问题处理不及时，影响项目进度及客户满意度。
  • 外部主要是随着房地产企业的发展和信息化进程，客户对信息化项目提出更高的要求，公司的产品、实施的专业能力都面临更高的挑战，项目要想交付需要比前几年做的更多更好。

需要开发支持

实施遇到的问题从内部外部分两部分：

  • 公司产品组需要加强产品优化，提升稳定性，跟进市场需求变化，同时提升产品颜值；
  • 开发项目组需要更多的支持实施对项目提出的需求和BUG进行处理，从根本上转变对需求的认识；
  • 客户提出需求是正常的合理的，开发需要从专业技术角度同实施去沟通需求的解决方案，怎样把需求考虑的更周全设计的更合理，满足最好超出客户的需求期望，然后加快需求交付速度和质量。

与客户沟通的效率

实施因为在项目现场，同客户沟通都是面对面的，沟通效率还是蛮高的，基本上客户提出什么问题，我们都没能够清楚的理解并形成文档让客户进行确认。

1.3、新项目实施周期控制

实施周期长，实施人员无法脱身，导致一连串的问题：

资源紧张、项目延期、回款进度慢…


  • 新产品弱不经风，上线前就已存在BUG
  • 新产品自适配性差，衍生二次需求
  • 现场实施未做好引导，举凡不满之处皆成二次需求

解决方案

  • 研究院需对产品的自适配性做深入调查，满足大众口味
  • 实施同事做好与客户的沟通

1.4、定制项目成功率

  • 定制项目成本一般比常规项目高
  • 定制项目难度一般比常规项目大
  • 定制项目交付标准没有一个绝对的衡量

解决方案

  • 配置足够的资源
  • 客户指派专员全程参与
  • 对项目整体把握，分阶段交付

1.5、组织技术与业务培训

不论技术或业务的培训，频次直线下降！

技术日新月异，员工无所提升！

技术作为命脉所在，所谓齐心捐就的图书馆，轻率儿戏！

罔顾培训创新，支持形同施舍！

解决方案

  • 恢复培训制度，讲师有偿培训，讲师给予可观回报，以资激励，谋求精品
  • 支持创新试验，准予时间与资金
  • 派遣参加社会组织的技术讲座
  • 每月准予 ￥300 -￥500 购书经费，组织读书分享会

1.6、升级与打包

明显的问题是：

  1. 升级不及时，造成问题滞后
  2. 升级包会有遗漏脚本情况
  3. 开发修改滞后的问题所耗精力较多
  4. 长期以往，造成效率低下

解决方案

  • 实行需求跟踪制度
  • 实行专人专事
  • 问责制，出现问题要追查到底
  • 形成及时有效沟通制度，保证效率

2.1、工作量评估

需求测试工作量估算不准确

解决方案

  • 通过需求文档、原型图，进行需求了解
  • 负责产品的研究院人员，项目开始之前或项目启动，需要先召开需求澄清会议，对有疑问的功能点解答
  • 版本发布的需求需先确定，开发过程中需求变更重新估算测试工作量，并且变更范围控制在20%以内
  • 每个模块指定一个负责人，工作量估算由测试负责人、模块负责人、需求测试负责人同时估算，取估算平均值

2.2、与研究院、开发的对接

测试、开发及研究院之间的对接不够顺畅

  • 原型图中，需求很模糊，需要开发做设计后，需求才清晰
  • 版本内的需求多次变动，一直到版本归档前还在变
  • 研究院人员支持度不够，经常不在公司，需求实现经常不能及时沟通
  • 需求功能变更或归档需求变动，未通知版本内的人员
  • 开发需求时间不足，导致版本问题较多
  • 现产品不断的做新功能，历史问题未安排时间修复，并且每个版本还遗留很多问题

解决方案

  • 研究院人员需在版本开始前将原型图、需求清单列出，并且版本发布的需求已做确认
  • 研究院人员给出的原型图，必须要清晰，描述清楚
  • 研究院人员，项目开始之前或项目启动后，召开需求澄清会议，对有疑问的功能点解答
  • 整个版本周期内，研究院人员确保可以全程参与
  • 将问题修复排入版本计划中，除解决本次版本的问题外，还解决部分历史问题，待版本稳定后再归档

解决方案

  • 开发人员喜欢批量送测，导致需求测试都堆压到一起，测试时间上不足，建议按开发计划上每个需求完成时间内转测
  • 原型上的需求，开发未做，而未及时通知测试人员，导致测试设计无效，建议未实现的需求及细节点要提前通知，并且送测报告中需写清楚
  • 存在阻塞问题，测试无法继续进行，导致测试资源浪费，建议测试人员筛选预测试用例，开发按预测试用例执行，执行通过后再转测

解决方案

  • 开发或测试过程中，与研究院人员沟通变更的功能或需求，研究院人员及时邮件知会，并且修改原型及更新需求清单
  • 整个版本过程中，开发计划、测试计划、需求澄清、测试用例评审等都需要开发、测试、研究院人员同时参与，且多沟通
  • 版本归档前，研究院人员验收通过后，方可归档

2.3、测试团队梯队建设

测试团队梯队建设

解决方案

  • 每个模块指定一个负责人，其它员工对该模块功能不清楚，可随时咨询模块负责人
  • 给每次新员工，指定导师，导师进行一对一的指导
  • 老员工主动帮助新员工业务学习，新员工也需要多努力，多向老员工学习

解决方案

  • 员工在某一或某些领域比较擅长、精通，能发挥所长，并带动其它员工
  • 项目管理建设，培养测试人员项目管理方法，如何编制测试计划，并随项目进度适时调整，识别并规避风险
  • 可定制开展话题讨论会，建立知识共享库
  • 制定培训计划，组织业务及技能培训
  • 调动部门员工的积极性，多组织活动

3、绩效

绩效制度与福利

  • 3.1、产品与开发绑定
  • 3.2、积极性与幸福指数

3.1、产品与开发绑定

开发人员理解需求不深，和业务有点脱离，设计人员得到的支持也不够

  • 产品需求和开发脱离，成了2个团队，需求人员长期不在公司。
  • 产品经理，给予开发团队支持力度太小，往往就一次交底会议和一个非常粗略的原型图，连个文档都没有。

解决方案

  • 日常上班，产品经理要在一直在团队中，时刻关注项目本身的实现效果、时刻给大家培训和讲解业务需求。
  • 绩效考核，产品经理与整个团队绑在一起。
  • 产品经理和开发技术团队，归根结底，是属于一个团队，可以由产品经理来担任项目经理，开发组长可以承担设计和主力开发。专人干专事，一人干好一件事。

3.2、积极性与幸福指数

  • 长期赶工、加班的工作模式，无法让员工保持长久的激情。
  • 员工成长空间，不明朗。
  • 企业文化缺失，应该说，没有企业文化。
  • 薪酬，和行业相比，处于较低水平。

解决方案

  • 制度上

    根本上缓解长期赶工、加班的状况，周期式的项目，都会有一个阶段性的竣工的，有一个缓冲的时间。

  • 员工成长空间

    最好能给员工以希望，有可以参照学习的榜样，是行之有效可以达成的成长。

解决方案

  • 企业文化

    可以首先定一个公司文化的基调，人事部门和各部门领导都尽量按照这个基调来管理团队，体现这个公司文化。

  • 薪酬

    努力和行业标杆靠拢，靠不拢没关系，尽量摆一个努力靠拢的姿态。

4.3、工期与质量

工期

工期不是项目经理或开发人员评估的，是指定的。这个很不合理，在资源不匹配的情况下，指定一个不合理的时间，想不延期都难。

资源配备不足。一个合理的团队，除了项目经理之外，高中初级开发工程师和测试工程师，都应该有一个合理的配置。

需求分析时间过长、需求文档不足。经常被要求什么时候完成项目，而对什么时候开始需求分析、完成需求分析不做要求。将按时完成项目的压力，堆积到开发和测试阶段。

干扰、打断，无时无刻都有，频繁干扰主要来自实施、客服。

工期解决方案

工期，一定是要经过合理评估的，才予以实施，千万不要拍脑袋。

合理的团队人员配备上，除了项目经理之外，高中初开发和测试4种角色的配备，
应该保持在1:2:2:2（少于5人的团队，1:1:1:1），高级工程师一定不能少。如果工期要求严格，那么就要配备足够的资源，最终制定的工期一定要和评估的工作量人天是匹配的。

指定需求分析的标准（主要是文档标准），需求交底往往是会议沟通的形式，凭借大脑是无法记忆很多的东西的，一定要以文档的形式提交需求分析成果，而且一定要细致，能够基于它做详细设计。

需求分析人员，在项目开发前期，一定要长期驻留，特别是设计阶段，要随时提供支持。这个非常重要。

产品开发、项目维护，一定要职责分开、明确。实施和客服，尽量不要干扰非本项目群人员。

4.3、工期与质量

质量

赶工。纵观公司多年来所开发的产品，每一个产品都是赶出来的，未开发完（或者还未开发）就已经卖给客户了。需求分析、系统设计等在软件项目开发周期中很重要的一些环节，没有分配时间，完全被忽视了。

项目经理管理职责缺位。各个项目团队的项目经理或主管，都没有把应承担的职责担起来，他们经常在干的事情是：主力开发（团队中最主力的那个！）。项目经理做了程序员该做的事情，自己应该做的事情（进度和质量管控，尤其是质量管控），没做。

测试粒度和强度不够。测试在项目开发周期中，介入的太晚，目前主要是测试工程师向项目经理询问需求及实现，然后编写测试用例，在开发过程中或结束时，进行测试。由于项目经理的时间被开发工作所占据，没有时间去评审测试用例。一个产品，往往就只有一个测试，在项目开发阶段的尾声，一个测试，远远不够。

质量解决方案

制定可行的开发流程。从项目立项起，就要根据预估的项目体量，确定可行的项目周期（发布节点）。然后制定需求、设计、开发、测试、发布等各大时间点。各个阶段占整个项目工期的时间比重大约如下：

需求：不超过总工期的1/4； 设计：不超过总工期的1/5； 开发：不超过总工期的1/2

测试和debug：不超过总工期的1/3（测试和开发有重叠）。一定要严格管控这个时间，尤其是第一个时间点。

在制定项目总工期时，一定要考虑到项目的体量、既有项目的业务和技术的基础，还要参考技术人员的意见，不能拍脑袋说，1个月开发出来、2个月开发出来。

测试提前介入。需求确认、交底等评审会议，测试全部要参加。测试对业务的理解、软件模型的理解，应达到项目经理的水准。
